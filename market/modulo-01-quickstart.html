Los 8 pasos · La ascension productiva

8 PASOS · DEL UMBRAL AL PRIMER COMANDO PRODUCTIVO

  1.  cd c:\fintexa\{tunombre}              parado en carpeta git
  2.  claude                                  abrir sesion
  3.  /mcp                                    Atlassian -> reiniciar Claude
  4.  "leer de jira ticket {TID}"             validar MCP
  5.  /fguide Soy {ROL}                       coach del rol
  6.  /fhelp como se usa {COMANDO}            lookup comando
  7.  /fguide Soy {ROL} flujos {COMANDO}      flujos completos
  8.  /{COMANDO} {TID}                        primer comando productivo

MAPPING ROL -> COMANDO PASO 8

  QA Lead        /fsqa DoR {TID}          PM             /fticket {TID} --mode completo
  Dev Senior     /fdoc {TID}              Product Owner  /freq {TID}
  Tech Lead      /freview --full {TID}    SRE / Oncall   /fd-incident p0
  Analista       /freq {TID}              Soporte N1/N2  /fticket {TID} --mode forensic
  CISO           /fd-secscan              Frontend Arq   /fscaffold-frontend
  Dev Junior     /fscan src/{Servicio}    Capital Humano /fretro --full
  CEO/CFO/QSA    /fwarroom --preset=board
Si paso 8 falla con "no estoy en repo git" -> volver a paso 1 (causa #1 de friccion).

Statusline + diagnostico · El pulso de Fintexa

STATUSLINE · EL PULSO DEL SANCTUM

  [FintexaMarketDev] > v2026.504.2-beta · 45 skills · ctx 23%

  [branch]      git activa            [main] -> cambiar antes de editar
  vXXXX-beta    version marketplace   ? o vieja -> /fhelp actualizar marketplace
  45 skills     skills cargados       <40 -> reinstalar
  ctx XX%       contexto usado        >80% -> /fcompact YA

MANTENIMIENTO BASICO (3 comandos universales)

  /fhelp version                  abrir sesion · validar version + skills count
  /fhelp actualizar marketplace   update autonomo (lenguaje natural)
  /fcompact                       comprimir contexto + pendientes + recovery

SI ALGO FALLA · cascada de remedios

  Unknown skill                  -> /fhelp y listar comandos validos
  Comando no escribe artefacto   -> falta git init o git clone (paso 1)
  MCP Jira no responde           -> /mcp + cerrar/abrir Claude
  Version ? en /fhelp            -> /fhelp actualizar marketplace
  Comando lento o cuelga         -> /fcompact (ctx >70% o turnos >15)
  Statusline ausente             -> bash setup.sh + reiniciar
  Comandos nuevos no aparecen    -> cerrar + claude de nuevo
  Sesion: not logged in          -> claude /logout && claude /login
El statusline es el pulso de Fintexa. Cuatro signos vitales: rama, version, skills, y contexto. Regla unica: si el contexto pasa el ochenta por ciento, compactar ya. Tres comandos universales para mantenimiento: version, actualizar, compactar. Si la primera sesion termina aca, el cliente esta autonomo.

FINTEXA MARKETPLACE V2 · QUICKSTART Q04 · 3 min

Comandos minimos obligatorios · El umbral del merge

COMANDOS MINIMOS OBLIGATORIOS · A PARTIR DE AHORA

  Todo repo microservicio + toda PR deben cumplir el gate minimo.
  Sin estos dos commands -> rechazo automatico de pasaje a develop / staging / main.

GATE 1 · ENTRADA AL REPO

  /fscan src/{Servicio}    reverse engineering inicial del codigo
  /fdoc                    SDD 14 secciones — doc de referencia del servicio

  Sin /fdoc el repo NO se considera "documentado"
  Minimo: ejecutar /fscan + /fdoc al iniciar el repo en marketplace

GATE 2 · SALIDA AL MERGE

  /freview --full {TID}    review 101 items + dictamen APPROVE | REJECT

  Sin /freview --full APPROVE la PR NO mergea
  Aplica a develop, staging, main por igual — sin excepciones

RECOMENDADO

  /freq {TID}              entrevista + criterios de aceptacion
  /fspec {TID}             HANDOFF-CARD (contrato spec -> dev versionado)
  /fforensic               auditoria pre-implementacion + pre-review
Dos comandos no negociables. Gate uno: cada repo entra con scan mas doc. Gate dos: cada pull request pasa por review modo full con dictamen approve. Sin esto, la PR se rechaza automatico. Aplica a develop, staging y main por igual. En el medio, camino recomendado: req, spec, forensic. Los dos gates son la entrada y la salida del marketplace.
A partir de ahora, dos comandos no negociables. Gate uno: cada repo de microservicio entra al ecosistema con /fscan mas /fdoc — el doc generado por ingenieria reversa es el baseline del servicio. Sin eso, el repo no esta documentado. Gate dos: cada pull request a develop, staging o main pasa por /freview --full con dictamen APPROVE — sin esto, la PR se rechaza automaticamente. Aplica a los tres ambientes por igual. En el medio, hay un camino recomendado — /freq, /fspec, /fforensic — que eleva la calidad y evita retrabajo. Recomendado, no obligatorio. Pero los dos gates si — son la entrada y la salida del marketplace. Pregunta dirigida: el repo de tu equipo ya tiene /fdoc ejecutado? Si no, ese es el primer trabajo del lunes.
