💡 La Analogía del Libro

Un índice en base de datos es como el índice de un libro:

❌ Sin Índice (Table Scan):
Leer página por página hasta encontrar "Revolución Francesa" - Lento y tedioso
✅ Con Índice (Index Seek):
Ir al índice, encontrar "página 210" e ir directamente - Rápido y preciso

Clase 1: Fundamentos - Clustered vs Non-Clustered

📚 Índice Clustered

  • Solo UNO por tabla
  • Define el orden físico de los datos
  • La PRIMARY KEY crea automáticamente un Clustered Index
  • Es la tabla misma, ordenada
Analogía: Una guía telefónica ordenada alfabéticamente por apellido

📖 Índice Non-Clustered

  • Hasta 999 por tabla
  • Estructura separada con punteros
  • Contiene solo las columnas indexadas
  • Apunta al Clustered Index
Analogía: Índice temático al final del libro

🎯 Ejemplo Práctico: Tabla de Productos

Escenario: Millones de productos, búsquedas frecuentes por SKU

Creación de Índices en SQL Server

CREATE TABLE Products ( ProductID INT PRIMARY KEY, -- Índice Clustered automático ProductName NVARCHAR(200), SKU NVARCHAR(50) NOT NULL, Price DECIMAL(10, 2) ); -- Consulta LENTA sin índice SELECT ProductName, Price FROM Products WHERE SKU = 'ABC-12345'; -- Crear índice Non-Clustered CREATE NONCLUSTERED INDEX IX_Products_SKU ON Products(SKU); -- ¡Ahora la misma consulta es RÁPIDA!

❌ Sin Índice

Clustered Index Scan
Lee toda la tabla secuencialmente

✅ Con Índice

Index Seek
Búsqueda directa y precisa

Índices con Entity Framework Core

// En tu DbContext, dentro de OnModelCreating protected override void OnModelCreating(ModelBuilder modelBuilder) { modelBuilder.Entity<Product>(entity => { // EF Core sabe que esto será el índice Clustered entity.HasKey(p => p.ProductId); // Definir índice Non-Clustered entity.HasIndex(p => p.Sku, "IX_Products_SKU"); }); } // Consulta LINQ que se beneficia del índice var product = await context.Products .FirstOrDefaultAsync(p => p.Sku == "ABC-12345");

💡 Tip Pro

EF Core traduce automáticamente tu consulta LINQ a SQL con WHERE. Gracias al índice IX_Products_SKU, se convierte en un Index Seek ultra rápido.

Clase 2: El Arte del Orden - Índices Compuestos

🥇 Regla de Oro del Orden

Un índice en (ColA, ColB, ColC) puede usarse eficientemente por consultas que filtran por:

  • ✅ ColA
  • ✅ ColA y ColB
  • ✅ ColA, ColB y ColC
  • ❌ Solo ColB o solo ColC

📞 Analogía: Guía Telefónica Nacional

❌ Mal Orden: (Nombre, Apellido, Provincia)

Para buscar "Juan Pérez" en "Mendoza", debes revisar todos los "Juan" del país

✅ Buen Orden: (Provincia, Apellido, Nombre)

Vas directo a "Mendoza" → "Pérez" → "Juan"

Selectividad: El Arte de Filtrar Mejor

🎯 ¿Qué es Selectividad?

📈 Alta Selectividad (BUENA)

  • Email (único por usuario)
  • DNI/ID (único)
  • CustomerID (pocos valores repetidos)

📉 Baja Selectividad (CUIDADO)

  • Status ('Activo'/'Inactivo')
  • Género ('M'/'F'/'Otro')
  • Categoría (pocas opciones)

⚡ Principio de Diseño

Coloca PRIMERO la columna con MAYOR selectividad

-- Consulta frecuente: facturas pendientes de un cliente SELECT InvoiceID, Amount FROM Invoices WHERE CustomerID = @customer AND Status = 'Pendiente'; -- ❌ MALO: Baja selectividad primero CREATE INDEX IX_Invoices_Status_Customer ON Invoices(Status, CustomerID); -- ✅ BUENO: Alta selectividad primero CREATE INDEX IX_Invoices_Customer_Status ON Invoices(CustomerID, Status);

Clase 3: Optimizando ORDER BY

⚠️ El Problema del "Costly Sort"

Si tu consulta pide datos en un orden diferente al del índice, SQL debe hacer una operación SORT costosa en memoria o disco.

📅 Ejemplo: Últimas órdenes de un cliente

SELECT OrderID, OrderDate, Amount FROM Orders WHERE CustomerID = @customer ORDER BY OrderDate DESC; -- ⚠️ Índice aceptable pero no óptimo CREATE INDEX IX_Orders_Customer ON Orders(CustomerID); -- Acelera el WHERE pero aún necesita SORT -- ✅ Índice ÓPTIMO CREATE INDEX IX_Orders_Customer_Date ON Orders(CustomerID, OrderDate DESC); -- Elimina completamente la operación SORT

🎯 Resultado

SQL encuentra al cliente y lee las órdenes en el orden que ya están almacenadas. ¡La operación Sort desaparece del plan de ejecución!

El Santo Grial: Covering Indexes

🏆 ¿Qué es un Covering Index?

Un índice que contiene TODAS las columnas que la consulta necesita. SQL puede responder usando solo el índice, sin tocar la tabla principal.

📝 Ejemplo: Dashboard de Blog

SELECT Title, CreatedAt FROM BlogPosts WHERE Status = 'Published' ORDER BY CreatedAt DESC; -- ⚠️ Índice bueno pero incompleto CREATE INDEX IX_Blog_Status_Date ON BlogPosts(Status, CreatedAt DESC); -- Aún necesita "Key Lookup" para obtener Title -- 🏆 COVERING INDEX (ÓPTIMO) CREATE NONCLUSTERED INDEX IX_Blog_Covering_Dashboard ON BlogPosts(Status, CreatedAt DESC) INCLUDE (Title); -- ¡La magia está aquí!
// En EF Core modelBuilder.Entity<BlogPost>() .HasIndex(p => new { p.Status, p.CreatedAt }) .IncludeProperties(p => p.Title) // INCLUDE en EF Core .HasDatabaseName("IX_Blog_Covering_Dashboard");

Clase 4: Detección y Mantenimiento

🕵️ El Plan de Ejecución - Tu Detective

  • Table Scan / Clustered Index Scan con costo > 90% = 🚨 ALARMA
  • Index Seek = ✅ Todo bien
  • Missing Index (texto verde) = 💡 Sugerencia de SQL

💡 Tip Pro: Missing Index DMV

SQL Server registra internamente qué índices echa en falta. ¡Puedes consultarlo!

-- Consulta para encontrar los índices más necesitados SELECT migs.avg_total_user_cost * (migs.avg_user_impact / 100.0) * (migs.user_seeks + migs.user_scans) AS improvement_measure, 'CREATE INDEX [missing_index_' + CONVERT(varchar, mig.index_group_handle) + '_' + CONVERT(varchar, mid.index_handle) + '] ON ' + mid.statement + ' (' + ISNULL(mid.equality_columns,'') + CASE WHEN mid.equality_columns IS NOT NULL AND mid.inequality_columns IS NOT NULL THEN ',' ELSE '' END + ISNULL(mid.inequality_columns, '') + ')' + ISNULL(' INCLUDE (' + mid.included_columns + ')', '') AS create_index_statement FROM sys.dm_db_missing_index_groups mig INNER JOIN sys.dm_db_missing_index_group_stats migs ON migs.group_handle = mig.index_group_handle INNER JOIN sys.dm_db_missing_index_details mid ON mig.index_handle = mid.index_handle ORDER BY improvement_measure DESC;

Mantenimiento: La Higiene de los Índices

⚠️ Fragmentación

Con el tiempo, después de muchos INSERT, UPDATE y DELETE, los índices se fragmentan. El orden físico ya no es secuencial, ralentizando las lecturas.

🟢 < 5% Fragmentación

Acción: Ninguna
Estado: Óptimo

🟡 5-30% Fragmentación

Acción: REORGANIZE
Operación: En línea

🔴 > 30% Fragmentación

Acción: REBUILD
Operación: Puede bloquear

-- Revisar fragmentación SELECT OBJECT_NAME(ips.object_id) AS TableName, i.name AS IndexName, ips.avg_fragmentation_in_percent AS FragmentationPercent FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, 'LIMITED') ips INNER JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id WHERE ips.avg_fragmentation_in_percent > 5 ORDER BY ips.avg_fragmentation_in_percent DESC; -- Mantenimiento ALTER INDEX IX_Products_SKU ON Products REORGANIZE; -- Para 5-30% ALTER INDEX IX_Products_SKU ON Products REBUILD; -- Para >30%

Mejores Prácticas del Experto

✅ SÍ Hacer

  • Índices en Foreign Keys - Casi siempre beneficiosos para JOINs
  • Estadísticas actualizadas - UPDATE STATISTICS regularmente
  • Monitorear planes de ejecución - Tu herramienta principal
  • Usar DMVs - Déjate guiar por los datos
  • Covering indexes para consultas críticas

❌ NO Hacer

  • Sobre-indexar - Cada índice ralentiza escrituras
  • Crear índices "por si acaso" - Solo los que necesitas
  • Ignorar el orden en índices compuestos
  • INCLUDE columnas gigantes - Usa con moderación
  • Olvidar el mantenimiento - Fragmentación mata rendimiento

🎯 Regla de Oro

Mide, no adivines. Usa planes de ejecución, DMVs y herramientas de monitoreo para tomar decisiones basadas en datos reales, no en suposiciones.

💡 Tips del Arquitecto

  • Entorno de Staging: Prueba índices en datos reales antes de producción
  • Mantenimiento automatizado: Scripts de REORGANIZE/REBUILD programados
  • Monitoreo proactivo: Alertas por fragmentación alta o missing indexes

Casos de Uso Empresariales

🏦 Sistema Bancario: Transacciones

-- Consulta crítica: Movimientos de cuenta en rango de fechas SELECT TransactionID, Amount, Description, TransactionDate FROM Transactions WHERE AccountID = @accountId AND TransactionDate BETWEEN @startDate AND @endDate ORDER BY TransactionDate DESC; -- Índice óptimo CREATE NONCLUSTERED INDEX IX_Transactions_Account_Date_Covering ON Transactions(AccountID, TransactionDate DESC) INCLUDE (Amount, Description);

🛒 E-Commerce: Catálogo de Productos

-- Búsqueda por categoría, marca y precio SELECT ProductName, Price, ImageUrl FROM Products WHERE CategoryID = @category AND BrandID = @brand AND Price BETWEEN @minPrice AND @maxPrice AND IsActive = 1 ORDER BY Price ASC; -- Estrategia: Múltiples índices especializados CREATE INDEX IX_Products_Category_Brand_Price ON Products(CategoryID, BrandID, Price) WHERE IsActive = 1; -- Filtered Index

Performance Tuning Avanzado

🎯 Filtered Indexes

Índices que solo incluyen filas que cumplen una condición específica

-- Solo indexar productos activos (más eficiente) CREATE INDEX IX_Products_Category_Active ON Products(CategoryID) WHERE IsActive = 1; -- Solo indexar órdenes del último año CREATE INDEX IX_Orders_Customer_Recent ON Orders(CustomerID, OrderDate) WHERE OrderDate >= '2024-01-01';

📊 Columnstore Indexes

  • Para queries analíticos y reportes
  • Compresión extrema de datos
  • Ideal para Data Warehouses
CREATE CLUSTERED COLUMNSTORE INDEX CCI_SalesHistory ON SalesHistory;

🔍 Full-Text Indexes

  • Para búsquedas de texto completo
  • Soporte para sinónimos
  • Búsquedas fuzzy y ranking
CREATE FULLTEXT INDEX ON Articles (Title, Content) KEY INDEX PK_Articles;

📋 Consultas a Optimizar

Búsqueda de productos con filtros múltiples SELECT ProductID, Name, Price, Rating FROM Products WHERE CategoryID = @categoryId AND BrandID = @brandId AND Price BETWEEN @minPrice AND @maxPrice AND IsActive = 1 AND StockQuantity > 0 ORDER BY Rating DESC, Price ASC; -- 2. Historial de órdenes del usuario SELECT o.OrderID, o.OrderDate, o.TotalAmount, oi.ProductName, oi.Quantity, oi.UnitPrice FROM Orders o INNER JOIN OrderItems oi ON o.OrderID = oi.OrderID WHERE o.CustomerID = @customerId ORDER BY o.OrderDate DESC; -- 3.
